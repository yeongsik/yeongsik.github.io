# 포스트 제목

컴퓨터 밑바닥의 비밀 chapter 1.1 정리

## chapter 1.1 여러분이 프로그래밍 언어를 발명한다면

### 프로그래밍 언어
- 프로그래밍 언어는 프로그래머가 컴퓨터에게 명령을 내리는 도구
- CPU가 이해하는 방식은 01010101
- 초반 프로그래머는 CPU에게 명령을 하기 위해 천공카드를 이용

### 어셈블리어
- 인간의 언어를 CPU의 언어로 바꿔보자는 첫번째 시도, CPU가 실행하는 명령 패턴(가산, 점프 등 몇가지 명령어) 발견
- 기계어를 인간이 읽고 이해할 수 있는 단어와 대응시킴 (ADD, SUB 등)
- 0과 1에서 sub, add, mov, call 등으로 바뀜

### 저수준에서 고수준 계층의 추상화
- 여전히 어셈블리어는 저수준 언어 (저수준 언어란 모든 세부사항을 신경써야함)
- 물 한잔 주세요라는 추상적 표현을 컴퓨터가 이해하는 언어로 바꾸려면 아래와 같이 구체적인 모든 세부사항을 신경써야함
  - 오른 다리를 내딛는다
  - 멈춘다.
  - 왼쪽 다리를 내딛는다.
  - 컵을 찾는다
  - 컵을 집는다
  - 물을 따른다
  - 컵을 놓는다
- 인간은 추상적 표현에 익숙하기에 거리를 좁힐 방법이 필요. (어셈블리어는 아직 저수준에 머물고 있음)

### 고급 프로그래밍 언어의 시작
- CPU가 이해할 수 있는 세부사항 목록 (어셈블리 명령들)에는 규칙 또는 패턴이 존재 
  1. CPU 명령에는 단도직입적임 -> 문장(statement) 단위
  2. statement 중에는 특정 행동의 특정 상황에 따라 어떤 명령어를 실행하는지 선택이 필요 (if)
  3. statement 중에는 반복되는 행동을 여러번 실행하는 것도 필요 (for)
  4. 같음 명령어 인데 세부사항만 다른 패턴 -> 매개변수의 차이 -> 함수의 탄생
- 참 거짓 분기 처리에 들어갈 statement, 반복할 statement, 매개변수를 사용하는 statement가 존재
- statement 안에는 조건, 반복 , 또다른 statement 가 들어갈 수 있음
- 재귀적 구조로 이를 syntax tree(구문 트리)라고 부름
- 코드가 아무리 복잡해도 결과적으로 모두 구문(syntax)으로 귀결 됨

### 우수한 번역가: 컴파일러
- 구문트리로 만들어서 컴퓨터를 이해시켜라
- 컴파일러는 리프노드부터 기계명령어 번역 -> 부모 노드 적용 -> 전체 구조의 구체적인 명령어 번역
- 고급 프로그래밍 언어의 탄생 : 구문트리로 코드 작성 -> 컴파일러 번역 -> CPU 실행 

### 해석형 언어의 탄생
- 세상엔 여러개의 CPU 존재, CPU에 따라 기계 명령어 실행 방법이 다르다
- 프로그래머는 작성한 코드가 가능한 많은 플랫폼에서 실행되길 원함
- CPU마다 상응하는 시뮬레이션 프로그램이 있으면 코드 하나로 여러 플랫폼에서 사용 가능해짐
- 가상머신(인터프리터) 등장
